# Google Maps Places Platform — Design Spec

## Purpose

A monorepo platform for managing Google Maps saved places via natural language. Extends the existing `gmaps-sync` scraping tool into a multi-app system with a shared SQLite data layer, Google Places API enrichment, and vector embeddings for semantic search.

Three apps share a common core:

- **sync** — Pull saved places from Google Maps via Playwright scraping (refactored gmaps-sync)
- **discovery** — Find new places nearby via Google Places API, queryable by natural language (future)
- **push** — Write local mutations back to Google Maps via Playwright (future)

The LLM interaction model is Claude Code running raw SQLite queries against the database — no MCP server, no custom query layer.

## Database Schema

Single SQLite file at `~/.gmaps-sync/places.db`. No profiles — one database, one Google account.

### places

The universal place record. Shared across sync, discovery, and manual sources.

```sql
CREATE TABLE places (
  google_place_id    TEXT PRIMARY KEY,
  legacy_id          TEXT UNIQUE,
  name               TEXT NOT NULL,
  lat                REAL NOT NULL,
  lng                REAL NOT NULL,
  address            TEXT NOT NULL DEFAULT '',
  comment            TEXT,
  content_hash       TEXT NOT NULL,
  created_at         TEXT NOT NULL,
  updated_at         TEXT NOT NULL,

  -- Enrichment fields (nullable until enriched)
  rating             REAL,
  user_rating_count  INTEGER,
  price_level        INTEGER,
  primary_type       TEXT,
  types              TEXT,               -- JSON array of type strings
  editorial_summary  TEXT,
  reviews_text       TEXT,               -- concatenated review text
  generative_summary TEXT,

  -- Boolean service/amenity attributes (0/1)
  serves_breakfast   INTEGER,
  serves_lunch       INTEGER,
  serves_dinner      INTEGER,
  serves_brunch      INTEGER,
  serves_beer        INTEGER,
  serves_wine        INTEGER,
  serves_cocktails   INTEGER,
  serves_coffee      INTEGER,
  serves_dessert     INTEGER,
  serves_vegetarian_food INTEGER,
  outdoor_seating    INTEGER,
  live_music         INTEGER,
  good_for_children  INTEGER,
  good_for_groups    INTEGER,
  allows_dogs        INTEGER,
  dine_in            INTEGER,
  delivery           INTEGER,
  takeout            INTEGER,

  -- Other enrichment fields
  business_status    TEXT,
  website_uri        TEXT,
  phone_number       TEXT,

  -- Pipeline timestamps
  enriched_at        TEXT,
  embedded_at        TEXT
);
```

### pending_mutations

Tracks mutations that haven't been pushed to Google Maps yet. Pending mutations for a place block pull updates to that place.

```sql
CREATE TABLE pending_mutations (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  type          TEXT NOT NULL CHECK (type IN (
                    'add_place_to_list',
                    'remove_place_from_list',
                    'move_place_between_lists',
                    'rename_list',
                    'create_list',
                    'delete_list'
                )),
  status        TEXT NOT NULL DEFAULT 'pending' CHECK (status IN (
                    'pending', 'in_progress', 'pushed', 'failed'
                )),
  place_id      TEXT,
  list_id       TEXT,
  payload       TEXT NOT NULL DEFAULT '{}',
  group_id      TEXT,
  seq           INTEGER NOT NULL DEFAULT 0,
  retry_count   INTEGER NOT NULL DEFAULT 0,
  max_retries   INTEGER NOT NULL DEFAULT 3,
  last_error    TEXT,
  created_at    TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
  updated_at    TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
  pushed_at     TEXT
);

-- Pull-blocking query: which place IDs have pending mutations?
CREATE INDEX idx_pending_mutations_place_status
    ON pending_mutations (place_id, status)
    WHERE place_id IS NOT NULL AND status IN ('pending', 'in_progress');

-- CLI `places pending` command
CREATE INDEX idx_pending_mutations_status
    ON pending_mutations (status, created_at);

-- Push engine: next group to execute
CREATE INDEX idx_pending_mutations_group
    ON pending_mutations (group_id, seq)
    WHERE group_id IS NOT NULL AND status = 'pending';

-- List-scoped queries
CREATE INDEX idx_pending_mutations_list
    ON pending_mutations (list_id, status)
    WHERE list_id IS NOT NULL;
```

**Payload examples by mutation type:**

| type | place_id | list_id | payload |
|------|----------|---------|---------|
| `add_place_to_list` | `ChIJ...` | `list_abc` | `{"place_name": "Blue Bottle"}` |
| `remove_place_from_list` | `ChIJ...` | `list_abc` | `{"place_name": "Blue Bottle"}` |
| `move_place_between_lists` | `ChIJ...` | `list_target` | `{"from_list_id": "list_source", "place_name": "Blue Bottle"}` |
| `rename_list` | NULL | `list_abc` | `{"old_name": "Restaurants", "new_name": "Dining"}` |
| `create_list` | NULL | NULL | `{"name": "Coffee Shops"}` |
| `delete_list` | NULL | `list_abc` | `{"name": "Old List"}` |

### places_fts (FTS5)

Full-text search index on enriched text fields.

```sql
CREATE VIRTUAL TABLE places_fts USING fts5(
  name,
  address,
  editorial_summary,
  reviews_text,
  content=places,
  content_rowid=rowid
);
```

Kept in sync via triggers on the `places` table. Uses SQLite's implicit `rowid` (present on all tables without `WITHOUT ROWID`).

### places_vec (sqlite-vec)

Vector embeddings for semantic search.

```sql
CREATE VIRTUAL TABLE places_vec USING vec0(
  google_place_id TEXT PRIMARY KEY,
  embedding float[384]
);
```

## Place Identity

**Canonical key:** `google_place_id` in `ChIJ...` format, resolved during enrichment via the Google Places API.

**The `ChIJ...` ID is NOT available in the getlist response** (where places are parsed during sync). The getlist response only contains:

- Composite numeric ID pair at schema paths `[1][6][0]` and `[1][6][1]`
- `/g/...` place reference at `[1][7]` (not currently extracted)

**Resolution flow for synced places:**

1. Parser extracts composite numeric ID + `/g/...` place reference (add `placeRef` to `schema.json` at path `[1][7]`)
2. Place is inserted with `legacy_id` = composite ID, `google_place_id` = `legacy_id` as temporary placeholder
3. Post-pull enrichment calls Google Places API Find Place (name + coords) to resolve the real `google_place_id`
4. The `places` row is updated with the resolved `google_place_id`

**Discovery places** arrive with `google_place_id` already set from the API. No resolution needed.

**Deduplication:** When enrichment resolves a synced place's `google_place_id` and it matches an existing Discovery place, merge the records — keep both metadata table rows, update the single `places` row.

No migration from existing data. First run starts with an empty database.

## Enrichment Pipeline

Lives in `packages/core`. Single entry point: `enrichPlace(db, googlePlaceId | {name, lat, lng})`.

### What it does

1. Calls Google Places API (New) Place Details endpoint
2. Requests Advanced tier fields: `editorialSummary`, `reviews`, `rating`, `priceLevel`, `primaryType`, `types`, all boolean service/amenity attributes
3. If called with name + coords (pre-resolution), first calls Find Place to get the `google_place_id`, then fetches details
4. Writes all enrichment fields to the `places` row, sets `enriched_at`
5. Generates embedding from concatenated text and writes to `places_vec`, sets `embedded_at`

### When it runs

- **After each sync pull:** enrich any places where `enriched_at IS NULL`
- **Discovery results:** arrive pre-enriched from the API — only need embedding generation
- **Re-enrichment:** optionally re-enrich places where `enriched_at` is older than `enrichment.reEnrichAfterDays` (default: 30). Triggered via `places enrich --force`.

### Failure handling

- Places that can't be found via the API: `enriched_at` is set to current time but enrichment fields stay NULL. Prevents retry on every pull.
- `places enrich --force` resets `enriched_at` to NULL for all places, triggering a full re-enrichment pass.
- Unenriched places work fine for structured queries (name, address, list membership). They won't appear in FTS/semantic searches on enrichment-only fields.

### Cost

Google Places API Advanced tier: ~$0.02/request. 1,000 places = $20, covered by Google's $200/month free credit. Re-enrichment only needed for stale data.

## Embedding Pipeline

Also in `packages/core`, called as the final step of enrichment.

### Model

`all-MiniLM-L6-v2` via `onnxruntime-node`. 384-dimension output vectors. ~80MB model binary, runs fully offline.

### Input text construction

```
{name}. {primaryType}. {editorial_summary}. {reviews_text}.
{attribute_string: "outdoor seating, serves cocktails, live music, good for groups"}
```

Unenriched places get a minimal embedding from `{name}. {address}.` — functional but lower quality for semantic queries.

### Storage and querying

Vectors stored in `places_vec` virtual table via sqlite-vec. Semantic queries use cosine similarity:

```sql
SELECT p.name, p.address, vec_distance_cosine(v.embedding, ?) AS distance
FROM places_vec v
JOIN places p ON p.google_place_id = v.google_place_id
ORDER BY distance
LIMIT 10;
```

### Performance

~5-20ms per embedding on CPU. 1,000 places takes under 20 seconds for a full batch. Negligible overhead for incremental post-pull runs. Embeddings are generated immediately after enrichment — no separate batch step.

## Conflict Resolution

**Strategy:** Pending mutations block pull updates for affected places.

When the sync pull runs `applyDiff`, it first queries:

```sql
SELECT DISTINCT place_id FROM pending_mutations
WHERE place_id IS NOT NULL AND status IN ('pending', 'in_progress');
```

Any place with a pending mutation is skipped during the diff — its local state is preserved until the mutation is pushed to Google.

The `places pending` CLI command shows all queued mutations:

```sql
SELECT * FROM pending_mutations
WHERE status IN ('pending', 'in_progress')
ORDER BY created_at;
```

## CLI

Single binary from `packages/cli`: `places`.

### Commands

```
places init            # Create DB, run migrations, open browser for Google login
places pull            # Scrape Google Maps → SQLite → enrich → embed
places pull --headed   # Debug mode (visible browser)
places pull --force    # Bypass consecutive failure guard
places status          # Show sync state, place/list counts
places enrich          # Manually trigger enrichment for unenriched places
places enrich --force  # Re-enrich all places (resets enriched_at)
places pending         # Show queued mutations
places prune           # Hard-delete soft-deleted places
places discover        # (future) Nearby search via Places API
places push            # (future) Push pending mutations to Google Maps
```

The `gmaps-sync` CLI ceases to exist. `places` absorbs all its functionality.

**No dedicated query command.** The database is queried directly via Claude Code using `sqlite3 ~/.gmaps-sync/places.db "SELECT ..."`.

## Changes from Existing gmaps-sync

1. **Source of truth:** JSON files → SQLite
2. **Storage format:** one-file-per-place → relational tables
3. **Place identity:** composite numeric ID → `google_place_id` (ChIJ...) resolved via Places API
4. **List membership:** `lists: string[]` array on place → `place_lists` join table
5. **Profiles:** removed entirely (single Google account)
6. **Sync state:** `sync-state.json` → `sync_state` table
7. **Store class:** replaced by Drizzle ORM queries against SQLite
8. **CLI name:** `gmaps-sync` → `places`
9. **New capabilities:** enrichment, embeddings, FTS5, semantic search, mutation queue
10. **schema.json:** add `placeRef` field at path `[1][7]` for `/g/...` place reference extraction
